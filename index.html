🎲 Provably-Fair Roll Verifier

Recompute any roll yourself, right here in your browser. Nothing is sent anywhere — this page has no server and no outside scripts. Right-click → View Source to confirm.

Each roll is HMAC-SHA512(server seed, "client seed-nonce"). The server seed was committed (its SHA-256 shown) before you played, so it couldn't be changed after the fact. Your client seed is yours. So neither side can rig the outcome.

Tip: server seed + client seed stay the same for a whole session — bump the nonce and re-verify to check every roll in that session.

How does this prove the game is fair?

Cryptographic hashes are a one-way street: easy in one direction, impossible in the other. The bot uses that asymmetry to lock itself in before you play. Here's the full chain:

  1. Before you play — the bot shows you the SHA-256 hash of its secret server seed (run ,serverseed to get it). That hash is the commitment — a fingerprint of the seed. SHA-256 can't be reversed, so you learn nothing about the seed itself, but the bot is now locked in: any future seed that doesn't match this fingerprint is provably fake.
  2. You set your own client seed (,clientseed). It's public, but the bot can't predict or change it — so the bot can't pre-pick a server seed that gives you a bad outcome.
  3. Every roll is HMAC-SHA512(server seed, "{client seed}-{nonce}"). Both seeds and the nonce all go in; you need all three to compute the output, and the bot can't reverse the output to learn the inputs. The nonce counts up by one each roll, so every roll is unique.
  4. The math — the first 13 hex characters of that hash become a fraction from 0 to 1 (÷ 2^52), then scaled into the roll's range. Same math everywhere, no special cases.
  5. Afterwards the bot reveals the server seed (,verify). You hash it here — and this page checks: does SHA256(revealed seed) match the commitment from step 1? Does each roll recompute to what the bot showed you? If yes to both, the bot didn't cheat. It couldn't have, without breaking SHA-256 / SHA-512 — the same math protecting HTTPS, Bitcoin, and every signed software update on Earth.

The asymmetry, in numbers:
Forward (you verifying a roll): ~microseconds on any phone.
Backward (bot forging a fake seed that matches the commitment): roughly 2256 operations — more than every computer ever built running for the age of the universe. Physically impossible, not just "hard".

Envelope analogy: before the round, the bot seals the server seed in a tamper-evident envelope and shows you a photo of the envelope's serial number. You can't read what's inside the envelope from the photo, but if the bot tries to swap the envelope after the round, the serial won't match the photo — and you'll catch it instantly.

"Can't I peek at the seed mid-game to predict future rolls?"
No. Any command that reveals the server seed (.verify, ,verifylink) is gated by the busy_check — if you're in an active game, those commands refuse until the game ends. And every successful reveal rotates the seed first: the next roll uses a brand-new secret seed that nobody (not even the bot's process, until that round ends) has shown you. So leaked seeds can only ever reveal the past.
